<h1>🛡️ CloudGuard AI – Operating Model (Internal Platform)</h1>

<p>👉 <strong>Exactly how real enterprise platforms are built.</strong></p>

<hr>

<h3>📌 Platform Usage Model</h3>
<p>CloudGuard AI is implemented as an <strong>internal enterprise platform</strong> designed to manage one organization's cloud environment.</p>

<p>In this project:</p>
<ul>
    <li>The platform is configured for our own AWS account</li>
    <li>It integrates with our Jenkins pipelines</li>
    <li>It analyzes our logs and metrics</li>
</ul>

<p><strong>There is no external customer onboarding in this model.</strong></p>
<p>This mirrors how real internal DevOps & Security platforms are built in companies.</p>

<hr>

<h3>🏗️ Environment Setup (One-Time)</h3>
<p>Before users log in, the <strong>Admin (DevOps Lead)</strong> configures the environment once:</p>
<ul>
    <li>AWS account credentials (IAM role / access keys)</li>
    <li>Jenkins URL and API token</li>
    <li>Kubernetes cluster access (kubeconfig)</li>
    <li>Prometheus metrics endpoint</li>
</ul>

<p>⚠️ These credentials are:</p>
<ul>
    <li>Stored securely in backend</li>
    <li>Used by CloudGuard AI services</li>
    <li>Not entered by normal users</li>
</ul>

<p>After this setup, the platform runs continuously.</p>

<hr>

<h3>🔐 User Authentication Model</h3>
<p>All users log in using CloudGuard AI internal authentication.</p>
<ul>
    <li>Email + password</li>
    <li>JWT-based authentication</li>
    <li>Role-based access control (RBAC)</li>
</ul>

<p><strong>❌ Users do NOT log in with AWS accounts</strong></p>
<p><strong>❌ Users do NOT log in to Jenkins or Kubernetes</strong></p>
<p>CloudGuard AI acts as a trusted control layer.</p>

<hr>

<h2>👥 USER ROLES & WHAT THEY SEE (VERY IMPORTANT)</h2>

<h3>👑 ADMIN USER (DevOps Lead)</h3>

<h4>Purpose</h4>
<p>Overall control and visibility of the platform.</p>

<h4>What Admin sees on login:</h4>
<ul>
    <li>System-wide dashboard</li>
    <li>Cloud security status</li>
    <li>CI/CD pipeline overview</li>
    <li>Kubernetes cluster health</li>
    <li>ML anomaly summaries</li>
    <li>User & role management</li>
</ul>

<h4>What Admin can do:</h4>
<ul>
    <li>Configure cloud & DevOps integrations</li>
    <li>Create users and assign roles</li>
    <li>View all alerts and logs</li>
    <li>Approve integrations and changes</li>
</ul>

<p>👉 Admin has <strong>read + control access</strong> across the platform.</p>

<hr>

<h3>🚀 DEVOPS USER</h3>

<h4>Purpose</h4>
<p>Deploy, operate, and monitor applications.</p>

<h4>What DevOps sees on login:</h4>
<ul>
    <li>CI/CD pipeline status (Jenkins)</li>
    <li>Build and deployment history</li>
    <li>Kubernetes pod & service status</li>
    <li>Prometheus metrics (CPU, memory, pods)</li>
    <li>Deployment alerts</li>
</ul>

<h4>What DevOps can do:</h4>
<ul>
    <li>Trigger deployments (via Jenkins)</li>
    <li>Investigate deployment failures</li>
    <li>Respond to operational alerts</li>
</ul>

<p><strong>❌ DevOps cannot modify AWS security policies</strong></p>
<p><strong>❌ DevOps cannot change IAM rules</strong></p>

<p>👉 DevOps focuses only on <strong>delivery & reliability</strong>.</p>

<hr>

<h3>🛡️ SECURITY USER</h3>

<h4>Purpose</h4>
<p>Ensure cloud security and compliance.</p>

<h4>What Security sees on login:</h4>
<ul>
    <li>AWS misconfiguration alerts</li>
    <li>Public S3 detection</li>
    <li>Open security group alerts</li>
    <li>IAM policy violations</li>
    <li>Auto-remediation logs</li>
    <li>Audit trails</li>
</ul>

<h4>What Security can do:</h4>
<ul>
    <li>Review detected misconfigurations</li>
    <li>Verify auto-remediation actions</li>
    <li>Investigate security incidents</li>
    <li>Monitor compliance status</li>
</ul>

<p><strong>❌ Security cannot trigger deployments</strong></p>
<p><strong>❌ Security cannot access CI/CD controls</strong></p>

<p>👉 Security focuses only on <strong>cloud posture & risk</strong>.</p>

<hr>

<h3>🤖 ML USER (Optional / Read-Only)</h3>

<h4>Purpose</h4>
<p>Observe intelligence generated by the system.</p>

<h4>What ML user sees on login:</h4>
<ul>
    <li>Anomaly detection results</li>
    <li>Risk scoring trends</li>
    <li>Abnormal usage patterns</li>
    <li>Historical insights</li>
</ul>

<h4>What ML user can do:</h4>
<ul>
    <li>Analyze system behavior</li>
    <li>Validate ML outputs</li>
    <li>Improve models (offline)</li>
</ul>

<p><strong>❌ ML user cannot control infrastructure</strong></p>
<p><strong>❌ ML user cannot deploy or remediate</strong></p>

<p>👉 ML provides <strong>insights, not actions</strong>.</p>

<hr>

<h2>🔄 HOW THE PLATFORM WORKS (END-TO-END)</h2>

<h3>1️⃣ Platform runs continuously</h3>
<ul>
    <li>AWS Config checks cloud resources</li>
    <li>Jenkins pipelines run on code changes</li>
    <li>Kubernetes runs applications</li>
    <li>Prometheus collects metrics</li>
    <li>ML analyzes logs and metrics</li>
</ul>

<h3>2️⃣ Events are processed automatically</h3>
<ul>
    <li>Misconfiguration detected → auto-remediated</li>
    <li>Deployment fails → alert generated</li>
    <li>Metrics spike → anomaly flagged</li>
</ul>

<h3>3️⃣ Users log in to observe & act</h3>
<ul>
    <li>DevOps responds to deployment issues</li>
    <li>Security reviews cloud risks</li>
    <li>Admin oversees everything</li>
</ul>

<p><strong>Even if no user logs in, the platform keeps working.</strong></p>
<p>👉 CloudGuard AI is <strong>proactive, not reactive</strong>.</p>
